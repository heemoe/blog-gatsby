# PLANS.md

## 1. 什么时候必须建计划

以下任务默认不能直接一路做到底，必须先补一份计划文档：

- 跨多个目录、多个页面模板、多个组件的改动
- 同时涉及内容、样式、构建配置中的两个及以上层面
- 预计会明显占用较多 Context Limit
- 需求边界还不够清晰
- 存在潜在推理死循环或“修一个点引出一串连锁修复”的风险
- 需要分阶段交付、暂停后恢复、或给后续协作者接手

## 2. 默认存放位置

统一放在：

- `docs/plans/<yyyy-mm-dd>-<topic>.md`

例如：

- `docs/plans/2026-04-21-homepage-style-refresh.md`

## 3. 推荐模板

```md
# <任务标题>

## Goal
- 这次要解决什么问题

## Scope
- 本次明确包含什么
- 本次明确不包含什么

## Read First
- 开工前必须先读的文件

## Risks
- 已知风险
- 可能的边界不清晰点
- 是否存在 context 或推理循环风险

## Plan
1. 第一阶段做什么
2. 第二阶段做什么
3. 第三阶段做什么

## Verification
- 每个阶段要怎么验证

## Open Questions
- 还需要用户确认的事项

## Status Log
- 记录阶段推进、关键决策、阻塞点
```

## 4. 维护规则

- 范围变化了，要同步更新 `Scope`
- 开工前读过的关键文件，要落到 `Read First`
- 遇到新风险，要补到 `Risks`
- 每完成一个阶段，要更新 `Status Log`
- 如果中途暂停，计划文档要足够让下一个协作者继续接上

## 5. 对 Agent 的要求

- 计划不是形式化文档，必须真的指导后续执行
- 不能把关键假设只放在聊天里而不写进计划
- 如果判断不需要建计划，也应该在回应里说明原因
